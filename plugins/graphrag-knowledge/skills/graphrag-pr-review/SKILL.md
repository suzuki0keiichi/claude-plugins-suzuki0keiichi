# PR レビュー（実装後・横断軸 + File / AI 実施）

diff 時のレビュー。**目的は QA でなくコントローラビリティ**（バグ検出は AI が行レベルで既に強い。ここは
「変更が人間所有の概念層の枠を超えていないか」＋「枠の中で触るべき同胞 File を触り損ねていないか」を見る）。

共通の土台・CLI の呼び方・逆引きの骨格・概念デルタ観点・3段の強制度は
**`${CLAUDE_PLUGIN_ROOT}/references/graph-review-method.md` を必ず先に読む**こと。本 skill は実行手順の要約だけ。

## 入力

`$ARGUMENTS` に base/head（例 `main...HEAD`）があればそれを、無ければ現在の作業差分を対象にする。

## 手順（method §1.5 → §2 → §2.5 → §3）

0. **鮮度プリチェック**（method §1.5）: 逆引きの前に、diff 内の各 File がグラフに存在するか
   `evidence --types File --limit 1` で確認する。無い File があれば「グラフがこの diff を知らない（索引が古い）。
   先に増分 index/carve を回すか、盲点として冒頭に明記」を**所見の冒頭**に出す。
   §4 の binding 漏れ（紐付け漏れ）とは原因が違う（こちらは索引遅れ）。
1. **変更 File を取る**:
   ```sh
   git diff --name-only <base>...<head>      # 範囲指定が無ければ git diff --name-only
   ```
2. **逆引きで枠を組む**（method §2）: 各変更 File について `ask` / `evidence` で
   - 所属 **Stratum / Pocket / Vein**（着地点・ルーティング信号。例: UI Layer に変更が入った）
   - 統べる **Decision / Constraint / Risk / OperationalKnowledge**（governance 逆引き）
   - 関連 **RejectedOption / supersedes 連鎖**（過去経緯）
   を引く。`ask` は領域ごとに1発、連打しない。`supersedes` / `refines` 連鎖と state を確認して**現役末端で枠を組む**
   （state superseded の Decision を基準点にしない）。
3. **順引きで影響圏を突き合わせる**（method §2.5）: 変更 File を seed に `evidence --types File --limit 1 --neighbors 2` で
   影響圏（同胞 File・統べる方針の他の宛先）を展開し、**影響圏内なのに diff に無い File**（触り残しの疑い）を挙げる。
   候補は**必ず裏取りする**: 候補ごとに統べる Constraint/Risk を引き（File 直宛は per-候補引きでしか出ない）、
   **候補 File と diff の該当箇所を実際に読む**。グラフ内だけで結論づけない（読まずに「要確認」で済ませるのは禁止）。
   問いは二つ — 変更後も候補は規範・変更側と**整合したままか**（取り残し: 同じ手当て・追従・新規範への適合の置き去り）／
   変更が候補側の規範の前提を崩していないか（前提崩し・迂回）。
   裏取りの結果、規範・制約を破る（例: エラー処理が片パスにしか入っていない / 前提が崩れてセキュリティが緩む）なら
   **ACK 必須へ格上げ**して騒ぐ（method §3 格上げ規則）。機械的な全列挙はしない。
   候補が多い時は裏取りの読みを **subagent に並列委譲**してよい（method §2.5 委譲項。ペイロード規定・判定非委譲・
   書き込み禁止・会計不変をそのまま守る — 委譲は実行手段の追加であって義務の縮小ではない）。
4. **概念デルタを判定**（method §3）: 枠に対し変更が超えていないか。
   - 却下案再導入 / 層破り（import 縦走）/ Decision 暗黙の反故 / Vein 規範 vs 実体（例: エラー処理の log だけ追加）/
     governance 違反（確立済み方針の退行）/ scope creep。
   - **has_premise 逆引き**（反故にする Decision を前提にしている現役ノードの列挙 = 前提崩れの波及）/
     **reduces_risk の解除**（弱めた Decision が抑えていた Risk の再開）/ **OperationalKnowledge の再踏み**
     （既知のハマり・ワークアラウンドの無効化や再導入）。
   - diff に `.graphrag/carving.json` の変更が含まれる場合、**免除追加を findings に昇格して人間裁定**。
   - 観点リスト外の概念デルタに気付いたら「グラフ外所見」と同じ作法で明示して出す（黙殺しない）。
5. **3段で分類**（method §3）: ERROR（構造拒否・稀）/ ACK 必須（却下案再導入・層破り・Constraint 違反）/ advisory（判断系・網羅チェック）。
   **段は上限ではない** — 実害（壊れる・穴が開く・前提崩れ）と判断した所見は ACK 必須へ格上げ。規範ノードが無い純粋な欠陥も「グラフ外所見」と明示して赤帯に出す（黙殺しない）。
6. **枠の自己点検**（method §4）: 統べる知識ノードが出てこない領域は「binding 漏れの疑い」を明記。`carving-check` #9 / `edge-suggest-policy` を案内。黙って「方針なし」としない。順引き（手順3）で候補が一件も出ない時も同様に binding 漏れを疑う。

## 出力

- **ACK 必須を冒頭にまとめる**（赤信号。「止めて確認」を促す。拒否はしない）。
- **網羅チェックを独立セクションで出す**（影響圏 − diff、method §2.5）: 候補は伝播経路＋根拠ノード id＋**読んだ上での判定**付き。
  枠ノードごとに「展開 File 数 / 裏取りした候補数 / 落とした候補と理由」の**会計**を記す —
  「該当なし」は会計付きでしか書けない。既定は advisory 帯、規範・制約を破る触り残しは冒頭の ACK 帯にも載せる。
- **advisory を該当箇所に添える**。各所見に**根拠ノード id を必ず添える**（traceable）。
- 各所見は「枠を超える可能性。直すのはコードか、方針（グラフ）側を更新して意図変更を承認するかは人間が決める」形で、fix 方向の裁定を人間に残す。
- **解決の書き戻し**（method §5）: ACK 必須所見を人間が「意図変更を承認」で解決したら、承認された意図変更を mutation
  （Decision update / 方針転換レシピ / 新 RejectedOption）として提案し、graphrag-knowledge skill の書き戻しへ繋ぐ。
  グラフを更新しないと次回も同じ所見が再発する（レビューの狼少年化）。
- 行レベルのバグ探しは主対象でない（必要なら別途 AI コードレビューへ）と明記し、概念層に集中する。
  ただし**見つけてしまった確信度の高い欠陥は黙殺しない** — 「グラフ外所見」と明示して赤帯に出す。
