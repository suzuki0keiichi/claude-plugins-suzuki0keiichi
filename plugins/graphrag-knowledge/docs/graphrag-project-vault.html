GraphRAG Knowledge — Project Vault

project vault
時限イニシアチブの
ナレッジグラフ

vault は 2 種類ある。system vault（受動的にコード知識を蓄積する地層）と project vault（能動的にプロジェクト健全性を監視する時限の計器盤）。 この附録は後者のスキーマを記述する。

種別project（VAULT.md に schema: project）
ノード型16 種
エッジ型22 種（型ペア検証つき）
Deliverablesystem vault の住人（cross-vault ref で参照）

system vault との関係。 知識系ノード（Decision / RejectedOption / Risk / Constraint / Goal / OperationalKnowledge / Investigation / ConversationChunk）は system vault から継承する共通の 8 型。project vault はこれに 8 つのプロジェクト固有型を加えた 16 型で動く。 Deliverable は system vault の構造系ノードであり、project vault からは cross-vault ref で指す。

§ 01

16 のノード型

system vault から知識系 8 型を継承し、プロジェクト固有の 8 型を加える。

知識系（system vault と共通） プロジェクト固有

知識系（継承）

8 型— system vault と共通の蒸留知識
Decision採用した判断

選択肢のなかから選んだもの。なぜそう決めたかを description に。

RejectedOption却下した案

試して駄目だった案。再検討時に同じ失敗を防ぐ一級市民。

Constraint制約

不変の外部条件（法規・SLA・技術制限）。

Goal目的・到達点

プロジェクトの目的因。二層パターン推奨: vision Goal（active のまま）+ gate Goal（achieved / abandoned）を refines で接続。

Riskリスク

脅威。state を持たない。緩和は reduces_risk エッジで。「ブロック」は Risk + risks_in → Task で表現。

OperationalKnowledge運用知識

運用で学んだ知見。繰り返しプロジェクト（年次予算・年次イベント等）で特に輝く。

Investigation調査

state: "active" | "closed"。led_to で判断を生む。

ConversationChunk会話の断片

出所となる生の対話記録。蒸留知識が derived_from で辿る一次情報。

プロジェクト固有

8 型— 時限イニシアチブの監視に必要な追加型
Source外部情報源

URL + 鮮度を持つ外部情報源。属性: source_kind（document / event / regulation / incident）、url、fetched_at、staleness_threshold。

Theme横断テーマ

プロジェクト横断の関心事（コスト削減、基盤移行等）。system vault の Layer / Concern / Component に相当。単一プロジェクトでは不要なことが多い。

Stakeholder利害関係者

人物・チーム・外部組織。

Resource資源

人・予算・資産・時間。属性: category（people / budget / asset / time）。

Milestoneマイルストーン

時間軸上のチェックポイント。Goal を時間に接地させる。state: "planned" | "achieved" | "missed"。

Assumption前提・仮説

検証前の前提。certainty は必須フィールド（Established / Expected / Assumed / Speculative）。has_premise で仮説ツリーを構成。

Agreement合意・契約

外部とのコミットメント。状態は前進のみ。撤回は expire → 新規 exploring で。

Task作業単位

判断に関わる粒度の作業。Jira チケット粒度のものは入れない。

§ 02

22 のエッジ型

system vault の文法を拡張し、計画構造・利害関係者・横断テーマの関係を加える。

太字は system vault から拡張された from / to 型、または project vault 固有のエッジ。

出所

— 知識を一次情報に繋ぎとめる

  • derived_from蒸留知識 + GoalAssumption | Task→ConversationChunk | InvestigationSource

§ 03

状態語彙

state を持つ型と、その遷移の制約。Risk と Assumption は state を持たない。

型 許容される state
Investigation active | closed
Decision / OK superseded のみ（state なし = 現行）
Goal planned | active | achieved | abandoned
Agreement exploring | negotiating | signed | active | expired
Task planned | active | completed | cancelled
Milestone planned | achieved | missed

Risk は state を持たない。緩和は reduces_risk エッジで表現する。 Assumption も state を持たない。確信度は certainty 属性で管理する。

Assumption の certainty（必須フィールド）

Level 意味 使いどころ
Established 確認済みの事実 契約済み・実測済み・過去の実績
Expected 証拠に裏付けられた高い確信 過去パターン・口頭合意・強い兆候
Assumed 未検証の前提 合理的な推測。迷ったらこれ
Speculative 希望的観測 根拠なし。「やってみないと分からない」

certainty は Assumption ノードの必須フィールド。commit-mutation は未指定を reject する。 迷ったら Assumed。定期的に再評価: Assumed → Established（検証済み）or Assumed → Speculative（反証あり）。

§ 04

cross-vault ref

Deliverable は system vault の住人。project vault からは ref 構文で指す。

エッジの to フィールドに vault:<slug>/deliverable:<system>:<slug> 形式を書く。 vault: プレフィクス付きのターゲットは、ローカルの存在チェックと型ペア検証をスキップする。

// Task が特定の Deliverable を必要とする
{ "type": "requires", "from": "task:proj:integration", "to": "vault:platform-x/deliverable:platform-x:product-v2.0" }

// Goal が Deliverable を狙う
{ "type": "targets", "from": "goal:proj:demo-on-v4", "to": "vault:platform-x/deliverable:platform-x:product-v1.5" }

// Goal の前提が Deliverable の完成
{ "type": "has_premise", "from": "goal:proj:launch", "to": "vault:platform-x/deliverable:platform-x:product-v2.0" }

VAULT.md の設定。project vault の VAULT.md には schema: project と vault_slug が必要。 vault_slug_aliases でリネーム時の後方互換を確保できる。

§ 05

知的背景 — なぜこの構造か

既存の知的伝統から design constraint を抽出した。1:1 の対応表ではなく、設計判断の発想源として。

  • PMI / PMBOKプロジェクトの定義「独自の成果を生む一時的な取り組み」→ vault = 時限イニシアチブ。Goal が achieved / abandoned で vault が閉じる
  • OKRObjectives and Key ResultsObjective は方向、Key Result は判定基準 → Goal の二層パターン: vision Goal（active のまま）+ gate Goal（achieved / abandoned）を refines で接続
  • PMBOK リスクマネジメント確率 × 影響度 / エビデンス階層前提の確信度を段階で管理する → Assumption の certainty 4 段階（Established / Expected / Assumed / Speculative）
  • Normanアフォーダンス理論存在するアフォーダンスは使われる（誤用含む）→ 状態語彙の意図的削減。「あるからいけない」— 不要な状態を置かないことで誤用を防ぐ
  • Dijkstra関心の分離 (1974)モジュールを横断する関心事は独立して扱う → Theme をプロジェクト横断の aspect として切り出す
  • 状態機械理論不可逆遷移とイベントソーシング遷移を不可逆にすれば履歴が自然に残る → Agreement の前進のみ状態。撤回は expire → 新規 exploring で履歴を保全
  • ソフトウェアプロダクトラインClements & Northrop (2001)バリアントは線形に置き換わるのではなく共存する → Deliverable は「版」ではなく「製品モデル」。用途別に並立する

overview の §04 と同様、これらは対応表による正当化ではない。 「なぜこの型にこの制約を課したか」を辿れるようにする発想源の記録である。
