GraphRAG Knowledge — Field Guide

システムの知識を、
ドキュメントではなく
「掘れる地層」として残す。

AI エージェントは、毎回「初見」でシステムに挑まされています。graphrag-skill は、プロジェクトの 設計判断・却下した案・制約・リスク・運用知識を型付きの知識グラフとして蓄積し、 エージェントが作業の最初に「先に引いて」から動けるようにする──探索の運に頼らない足場を渡す仕組みです。 正本は Obsidian でそのまま読める Markdown の vault 一つ。外部 DB はありません。

正本vault（Markdown + git）
ノード型13 種
エッジ型15 種（型ペア検証つき）
読み口ask（semantic + lexical）

先に答えておくと──Claude Code や Codex のメモリー機能が完璧に進化したら、この仕組みはお役御免になります。 これは LLM 製品がまだ持っていない「構造化された長期記憶」を外付けする繋ぎであり、その日が来るまでの実利を取りに行くものです。

§ 01

背景 — なぜグラフでナレッジを持つのか

初見で問題を解かされるエージェントに、「それっぽさ」の運ではなく、構造の足場を渡す。

AI エージェントは、毎回情報ゼロの状態から「初見」で問題を解かされるという、人間の開発者なら誰も引き受けない 不利な条件で働いています。この不利をどう埋めるかは段階的に進歩してきました──ドキュメントを読ませる、RAG で引く、 そして現在の主流である「エージェントが自ら探しに行く」エージェンティックサーチへ。それでも、まだ埋まらない穴があります。

第 1 段階
ドキュメント整備

人が書き、エージェントに読ませる。

限界: 書かれない・腐る・量が増えると「どれを読ませるか」自体が問題になる。

第 2 段階
RAG

類似度検索でチャンクを引き、コンテキストに足す。

限界: 引けるのは「似た文章」まで。関係は辿れず、網羅も担えない。

第 3 段階・現在の主流
エージェンティックサーチ

エージェント自身が ls・grep・read でコードベースを自走探索する。

限界: 「それっぽさ」頼みの探索。網羅できたかを誰も保証しない。

graphrag の立場
知識グラフ

判断・制約・関係を型付きのグラフ構造で保存し、semantic に引いてエッジを辿る。

狙い: 網羅を探索の腕前ではなく、データ構造の性質にする。

エージェンティックサーチは強力ですが、その正体は「それっぽいフォルダを ls し、それっぽい語を grep する」という ヒューリスティックな探索です。よく整理されたコードベースなら高い確率で当たります。しかし、それは保証ではありません。

🎲

網羅が保証されない

影響範囲を調べ尽くしたかどうかは、キーワード選びの運に懸かっている。命名・フォルダ構成・変数名が荒れたコードベースでは、「それっぽい場所」がそもそも存在しない。

🕳️

コードに書かれていない情報

過去に選ばなかった選択肢、ノウハウ、制約は、ファイルにはほとんど残っていない。どれだけ上手に探しても、grep すべき対象が最初から存在しない。

📚

ドキュメントでも覆えない

では文書化すればよいかというと、大量のドキュメントをキーワードだけで探すのは言い換えに弱く、すべてをコンテキストに積むのは物理的に不可能。

🧵

「波及」を問える構造がない

「ここを変えると何に効くか」という関係そのものは、コードにもドキュメントにも明示的には記録されない。読み手が毎回、推測で再構築している。

だから、知識を最初からグラフ構造で保存しておく──これが graphrag-skill の答えです。 判断・却下案・制約・リスクを型付きノードとして持ち、「何に効くか」「何を上書きしたか」を型付きエッジで持つ。 引くときは semantic 検索で雑な問いから入り、当たったノードからエッジを辿って影響圏を機械的に列挙する。 網羅性が、探索の腕前ではなくデータ構造の性質になります。

GraphRAG の弱点は更新コストです。知識を蒸留し、型を付け、関係を張って書き戻す手間は、平文のメモより明らかに重い。 それでも採る価値がある、と判断した根拠は開発のボトルネックの移動にあります。 コーディングエージェントの登場で設計・実装は桁違いに速くなり、いま詰まっているのはレビューと影響調査のほうです。

開発が速くなった分だけ、待ち行列はレビューと影響調査に集中する。判断の系譜と影響圏を構造で渡せれば、ここが軽くなる──グラフの更新コストは、この場所で回収する。

もう一つ、更新の重さ自体もスキル化で削っています。グラフへの書き戻しは、後でまとめてやる文書化作業ではなく、 エージェントとのやり取りの中に埋め込まれています。「○○することにする」と決まった瞬間、試した案が駄目だと分かった瞬間に、 会話の流れのままエージェントが型を付けて書き戻す（§08 の Proactive Persistence）。 普段なら言語化されずに消えていく暗黙知までグラフに乗りやすくなり、 使い込むほどエージェントは「このシステムの事情を知っているベテランエンジニア」に近づいていきます。

名前についての断り書き。「GraphRAG」は本来 Microsoft 発の手法名で、本家の定義はここでの説明と異なります ──文書をチャンクに区切り、LLM がエンティティと関係を抽出してグラフ化し、グラフ上の距離が近い塊を「コミュニティ」として 検出して要約を付ける、という自動構築のパイプラインです。このスキルは名前を借りつつ、コミュニティを事後的に 発見する代わりに、型付きスキーマで最初から構造を彫っておく我流の亜種です。コミュニティ要約が担う「塊の意味」は Layer / Concern / Component とその description が担っているので、看板に偽りあり、とまでは言わないでいただけると幸いです。 なお、コミュニティ方式をあえて採らなかった積極的な理由（構造ドリフトの検知）は §04 で述べます。

§ 02

意図 — 動かさない設計境界

このスキルの価値は型の列挙ではなく、切り分けの判断とそれを支える不変条件にある。

  • vault が単一正本 知識は vault（Markdown）に格納し、検索・索引・書き込みは全て vault を読む。外部 graph DB は v3 で撤去済み。graph.json は索引器の中間表現であって正本ではない。バックアップと履歴は git が担う。
  • LLM に生クエリを書かせない LLM が触れる面は「ランク済み JSON を読む」「型付きの追加コマンド／mutation plan を出す」の 2 つだけ。Cypher のような生クエリ層を露出しないことで、検証可能な読み書き境界を保つ。
  • semantic は非交渉 検索は lexical（一致・カバー率）と semantic（embedding cosine）を同じ重みで合算する。ベクトル索引が無ければ ask は黙ってフォールバックせず、明示エラーで停止する。品質の落ちた検索を一級経路にしない。
  • 出所必須（source backing） Decision / RejectedOption / Risk / 運用知識のような蒸留知識は、生の一次情報（ファイル・会話・調査）へ辿れることが schema 上の必須要件。出所のない「きれいな結論」は validation で reject される。
  • 却下案は一級市民 「なぜ採らなかったか」は最も腐りやすく最も失われやすい情報なので、RejectedOption を独立ノード型にし、再検討時に同じ議論を繰り返さないようにする。
  • 書き込みは常に検証つき原子操作 vault は手編集しない。add-* / commit-mutation が lock → 楽観排他（OCC）→ schema 検証 → 重複ゲート（新規ノードを既存の同型ノードと embedding 照合し、近すぎれば確認を要求）→ 原子 delta 書込 → ベクトル索引更新 → git commit を一括で担保する。失敗時は all-or-nothing でロールバック。複数エージェントの並行書込を前提にした作りになっている。
§ 03

全体像 — 知識はどう流れるか

作業の最上流で「引く」、一段落で「書き戻す」。その間に vault がある。

読み・書き・索引のすべてが vault を向く。エージェントの読み口は ask、書き口は add-* / commit-mutation のみ。
人間は同じ vault を Obsidian で開けばよく、エクスポートや同期は存在しない。

§ 04

グラフ種別 ① — 13 のノード型

3 つの系（構造・知識・横断）に分かれる。構造系は File + Deliverable、横断系は Layer / Concern / Component の 3 軸。

構造系（コードの実体） 知識系（蒸留された判断と経緯） 横断系（構造から独立に生きる意図）

構造系

2 型— 索引が機械的に生成する、グラフと実コードの接地面
Fileファイル

索引されたソースファイル。embedding の主担体で、シンボルや import は独立ノードにせず File の配列フィールドに持つ（ノード爆発の回避）。要約を機械テンプレから「意味」の要約へ書き換えることが検索品質の主レバー。

Deliverable成果物

リリースの産出物。並列に複数存在可能。project vault からは cross-vault ref で参照される。major 世代交代のみ supersedes を使う。

知識系

8 型— 会話と作業から蒸留される、このスキルの本体
Decision採用した判断

採用した設計判断。なぜそう決めたかを description に書く。supersedes で却下案を、refines で旧判断を継ぐ。覆された判断は state: "superseded" のまま残り（系譜保存）、検索では減点される。

RejectedOption却下した案

一級市民。試して駄目だった案とその失敗モードを独立ノードで残す。再検討時に同じ議論・同じ失敗を繰り返さないための、このスキーマで最も特徴的な型。

Constraint制約

守るべき制約。constrains で判断・ファイル・運用知識を縛る。「やってはいけない」を構造で持つ。

Goal目的・到達点

システムの目的因。refines で上位 Goal を細分化する。判断が「何のためか」を遡るための錨。

Riskリスク

将来踏みそうな、再利用されるリスク。risks_in で対象を指し、reduces_risk で緩和する判断と結ばれる。

OperationalKnowledge運用知識

再利用される運用知見・ワークアラウンド・ハマりどころ。CLI では add-ok で書く。

Investigation調査

進行中／完了した調査エピソード。focus 継続（前回の続き）の主語であり、led_to で判断を生む流れを残す。state: "active" | "closed" を持ち、resume が拾うのは active のもの。

ConversationChunk会話の断片

出所となる生の会話メモ。蒸留知識が derived_from で辿り着く一次情報。「なぜ」を後から検証する材料置き場。

横断系

3 型— ファイル構造が変わっても生き残るべきアーキテクチャの意図
Layer層

水平に積もる層＝アーキテクチャ層。「どの層か」（縦の位置）を持つ。

Component部品

局所に凝集した塊＝パッケージ／モジュールのまとまり。「どこにあるか」（構造）を持つ。

Concern横断関心

層も部品も貫いて走る筋＝横断的関心事。「何を横断的に気にしているか」を持ち、構造にも層にも縛られない。

なぜ 3 軸か — Layer ≠ Concern ≠ Component

3 軸を 1 つの型に潰すと、ファイル移動やリファクタのたびにアーキテクチャの意図が構造と心中します。 Layer（どの層か）・Component（どこにあるか）・Concern（何を横断的に気にしているか）は 互いに独立した切り分け軸です。3 種はいずれも evidenced_by で File を指し、 意図を必ず実体に接地させます。ファイルが動いても、層・部品・関心は生き残ります。

横断 3 種はどれも evidenced_by で File に接地する。ファイルが移動しても層・部品・関心のノードは残り、
「この関心はどこに効いているか」を File 側を張り替えるだけで追従できる。

なぜ型を「先に」固定するのか — ドリフトの基準点

本家 GraphRAG のコミュニティ検出は、「いまのデータにとって、ちょうどいい塊」をその都度作り直します。 これは適応力ですが、裏返すと──バイブコーディングで構造が当初のコンセプトから少しずつズレていっても、 塊のほうがズレに付いていくため、誰も検知できないということです。 型とエッジ文法を先に彫っておくと、意図した形が測量の基準点のようにグラフに残り、ズレは基準からの差分として 見えるようになります。形を変えること自体は禁じません。ただし「黙って馴染む」のではなく、 人間が気づき、確認してから変えるという手順に変わる。 だから大事なのは、この型が完璧か・ベストかであることよりも、まず型があること自体なのです。

では、この 13 型は「王道」なのか

我流です。設計にあたって「知識の型の、定番とされる分類」を哲学・認知科学側まで探しに行きましたが、 見つかったのは「鉄板の発想源は数あれど、王道──誰も異を唱えない単一の基盤──は無い」という事実でした。 分類は常に、それを立てた問いと用途のために組まれてきたものです。 ただし我流は思いつきと同じではありません。この 13 型は、確立した知的伝統からそれぞれ design constraint を一つずつ抽出して組み上げています。 また、結果として到達した設計は、独立した文脈で近い方向を向いています。 vault のファイル形式は Google が 2026 年に提唱した OKF (Open Knowledge Format) v0.1 の適合条件をそのまま満たしており（§09）、 「使うたびに知識が構造化されて蓄積する」という学習ループの設計は Satya Nadella が論じた Token Capital（§10）の考え方と本質的に重なります。 狙って合わせたわけではなく、同じ課題を真剣に追った結果の合流です。

  • カント『純粋理性批判』純粋悟性概念のカテゴリー表「否定」は実在と同格の一級カテゴリーである → 却下も判断と同格に。RejectedOption を独立ノードにする
  • Tulvingエピソード記憶と意味記憶の二系統経緯の記憶と結論の記憶は別系統 → ConversationChunk・Investigation（経緯）と Decision など（結論）を両建てで外部化する
  • Bartlett再構成的記憶記憶は思い出すたびに作り替えられる → 再構成バイアスから守る外部の正本 = vault を置く
  • Hutchins分散認知認知は頭の中だけでなく道具や記録物に宿る → vault をエージェントの認知の物理基盤として扱う
  • PROV-O来歴記述のオントロジー出所は独立した関係として明示する → documented_by / derived_from（出所必須コントラクト）
  • Toulmin議論のモデル主張は根拠と前提で支える → evidence / has_premise

一方で、「カントの何々がこのノード型に対応する」といった対応表による正当化はしません。 13 のノードから都合のよい 3 つを選ぶ組合せは 286 通りあり、後付けの対応表は 「権威に合わせて切り貼りしただけでは」という批判に耐えないからです。哲学はあくまで発想源に位置づけ、 型の正式な定義は「概念名と、適用をどう判定するかの手続き」で書く。そして分類の正しさは権威ではなく、 次章のエッジ文法と validateGraph による検証可能性で支えます。

ID 規約

すべてのノードは <typeSlug>:<system>:<slug> という ID を持ちます（例: decision:graphrag:vault-single-source）。<system> は ID の名前空間ラベルで、 System というノードが作られるわけではありません。スコープそのものは vault の境界が担います（vault = scope）。

§ 05

グラフ種別 ② — 15 のエッジ型と「文法」

何が何に繋がってよいかは型ペア規則で強制される。これがグラフのスープ化を防ぐ一貫性契約。

エッジは自由なリンクではありません。validateGraph が from / to のノード型の組を検査し、 規則外のエッジは書き込み時に reject されます。役割で 4 群に分けて読むと覚えやすい構成です。

なお、v3.3 で唯一の「整理用エッジ」だった contains は撤去されました。所属の情報は vault のフォルダ構造と ID 規約がすでに持っているためで、グラフに残るエッジは意味関係のみです。同様に、シンボルや呼び出し関係は 独立ノードにしません（ノード数の爆発と精度低下を避け、File の配列フィールドと要約に織り込む設計です）。

§ 06

読み方 ① — エージェントは ask で引く

grep でも glob でもなく、まず ask。1 コマンドで必要な深さまで自動で潜る。

エージェント（または CLI を叩く人間）の読み口は ask 一つです。雑な自然文の質問を投げると、 内部で retrieval ladder を自動で登り、必要な深さの結果だけを返します。 検索順位は lexical（完全一致・部分一致・単語カバー率）と semantic（embedding の cosine）を 同じ重みで合算したものです。

$ node --experimental-strip-types graphrag/cli.ts ask "なぜ vault が単一正本なのか" $ node --experimental-strip-types graphrag/cli.ts ask "認証まわりを触ると何に波及する?" --limit 8
stage 1
brief

概要応答。確信度が高い一致があればここで止まる（最小コスト）。

stage 2
search

ランク済みの近傍展開。ヒットの周辺ノードをエッジ伝いに広げる。

stage 3
evidence

出所付きの answer packet。どのノード・どのファイルが根拠かまで返す。

「無い」の判定と打ち切り — 連打しないための出力フィールド

final_stage — どこまで自動段上げしたか。evidence まで行って空なら本当に無い。一時的な調査ギャップであり、慌てて新規ノードを作る理由にはならない。
match_confidence — high なら採用。low / none なら段上げ済みなので、別キーワードを 1 度だけ試す。連打しない。
repeat_state — 同じ問いの繰り返しを launcher が数えている。excessive（3 回以上）になったらグラフ検索を打ち切り、コード・ドキュメントの直読みに移る。

前提が欠けていると ask は黙ってフォールバックせず大声で止まります。 vault が見つからない、embedding endpoint に届かない、ベクトル索引が無い──いずれも明示エラーです。 「検索精度が落ちた状態で動き続けて、静かに間違った答えを返す」ことを設計として禁じているためです。

§ 08

書き方 — 何を、いつ、どう永続するか

「覚えて」と言われるのを待たない。一段落したら能動的に書き戻す。

単発の知識は add-decision / add-rejected-option / add-risk / add-ok / add-investigation / add-constraint / add-goal という型付きコマンドで、引数だけで書けます。エッジも --premise（前提）や --constrains（制約対象）等のフラグで同時に張れ、 --aliases で検索用の別名（日本語の一般語 + 英語のコード語）を積めます。 複数ノード・エッジの張り替えを伴う複雑な変更は mutation plan（JSON）を commit-mutation に渡します。 どちらも書き込み前に 既存ノードの重複確認を必ず先に行い、新規よりも skip / update / supersede を優先します（書き込み時にも重複ゲートが embedding 照合で最後の網を張りますが、事前確認の代わりにはなりません）。

# 採用判断を 1 件永続化（--evidence で File へ接地。schema 上必須） $ node --experimental-strip-types graphrag/cli.ts add-decision \ --system myapp --slug session-store-redis \ --title "セッションは Redis に置く" \ --summary "sticky session を不要にするため外部化" \ --evidence file:myapp:src/auth/session.ts # 失敗した試みを記録（これが最も価値のある書き込み） $ node --experimental-strip-types graphrag/cli.ts add-rejected-option \ --system myapp --slug session-in-memory \ --title "インメモリセッション" \ --summary "ローリングデプロイで全ユーザーがログアウトする" \ --rejected-in-favor-of decision:myapp:session-store-redis

書き込みはすべて、次のパイプラインを通って vault に届きます。途中で失敗すれば all-or-nothing でロールバックされ、壊れた状態は公開されません。

lock→ OCC (楽観排他)→ schema 検証→ 重複ゲート→ 原子 delta 書込→ vector 索引更新→ git commit

書き込みの出力には提案（suggestions）が同梱されます — 「この判断はこの File を統べていそう」という binding 候補、よく似た既存判断との関係候補、 進行中の調査への led_to 候補、直前に読んだ判断の前提（has_premise）候補。 提案はどれも自動では張られません。採るか・理由を持って見送るかはその場の判断で、 bind されていない知識ノードの総数（binding_debt）は常に会計として見えます。 昔の vault にこの提案を一括で当て直す入口は edge-suggest-policy です。

🪨 永続するもの — 地層に残す
  • 採用した設計判断とその理由
  • 試して駄目だった案と失敗モード（特に拾う。書かないと同じ失敗を繰り返す）
  • 守るべき制約・将来踏みそうなリスク
  • 運用のハマりどころ・ワークアラウンド
  • 調査の経緯（何を調べ、何が判断に繋がったか）
🍃 永続しないもの — 風化に任せる
  • session 内でしか使わない一時的な探索メモ
  • 数分で戻せる scratch（変数名いじり、lint 通しの一時変更）
  • その場限りの観察・未確定の推測
  • コードや git 履歴を読めば分かること

書くタイミングは「ユーザーが結論を述べたとき」だけではありません。実装や修正が一段落した・コミットする直前という 無言の行為もトリガーです（Proactive Persistence）。その変更の背後にある判断・捨てた代替案・踏んだリスクを、 依頼を待たずに書き戻します。逆に、グラフと実コードの食い違いに偶然気づいたときは、勝手に書き換えず 選択肢（update / delete / skip / investigate）を提示してユーザーの判断を仰ぎます（Drift Reconciliation）。

グラフを背骨にしたレビュー（姉妹スキル）

読み書きの土台の上に、グラフと照合する 3 つのレビュー用スキルが載っています。目的は QA ではなく コントローラビリティ──「大枠はおまかせ、でも過去の判断と制約の枠は超えるな」を機械的に支えることです。

graphrag-design-review

実装前の設計案を、過去の Decision・Constraint・Goal と照合してレビューする。

graphrag-pr-review

PR / diff をグラフと照合し、概念レベルのズレ（方針違反・暗黙の影響圏）を検出する。

graphrag-review-doc

人間レビュアー向けに、変更を概念レベルで説明する HTML 資料を生成する。

§ 09

OKF (Open Knowledge Format) との関係

フォーマットの互換性と、フォーマットの外にある仕組みの話。

2026 年、Google が Open Knowledge Format (OKF) v0.1 を提唱しました。「YAML frontmatter 付き Markdown ファイルのディレクトリを git で管理する」という LLM Wiki パターンを、 ベンダー中立なフォーマット標準として定義するものです。

vault は OKF 適合

OKF v0.1 の適合条件は 3 つだけです。 graphrag-skill の vault はそのすべてを満たしており、変換やエクスポートなしでそのまま OKF v0.1 に適合しています。

条件 ① 予約名以外の .md に解析可能な YAML frontmatter がある 全ノードファイルに frontmatter あり ✓
条件 ② frontmatter に空でない type フィールドがある type は schema 上の必須フィールド ✓
条件 ③ 予約ファイル（index.md / log.md）が存在する場合は規定構造に従う 予約ファイルを使用していない ✓
→
3 / 3 適合 vault はそのまま
OKF v0.1 に適合

フォーマットの外にあるもの

graphrag-skill が提供しているのは、フォーマットではなく仕組みです。 意味を抽出すること、意味と意味を型付きエッジでつなぐこと、それを人間が管理可能な形式に固定すること、 そしてその一連の流れをスキルとして自動化すること──これらはすべてフォーマットの上位に位置する層であり、 OKF の仕様範囲には含まれていません。

層 OKF v0.1 graphrag-skill
ファイル形式 YAML frontmatter + Markdown 同じ（OKF 適合）
型の定義 type は自由文字列。型の一覧や意味は未定義 13 種の型付きノード。型ごとに必須フィールドと検証規則
関係の表現 Markdown リンクの地の文。関係の種類は機械的に取れない 15 種の型付きエッジ。型ペア検証つき
整合性の検証 壊れたリンクも許容。未知フィールドは無視 validateGraph で schema 検証、source backing 必須
書き込みの制御 規定なし lock → OCC → 検証 → 重複ゲート → 原子 delta → 索引 → git commit
検索 規定なし（消費者の実装次第） semantic + lexical 合算。retrieval ladder で段上げ

つまり、OKF は vault が既に満たしているファイル形式の層を標準化したものであり、 graphrag-skill がやっていることの大部分はその上の層にあります。 フォーマットが共通であること自体は良いことですが、 同じフォーマットの知識が実際に相互運用できるかどうかは、型・関係・検証の取り決めに依存します。

現時点では、OKF はフォーマット仕様のみの公開です。 型の体系・型付きエッジの検索・検証つき書き込みといった、知識を実際に運用するための層を担うツール群はまだ存在しません。 graphrag-skill はその層を既に持っているため、いま OKF に乗り換える理由はありません。 フォーマットの互換性は既にあるので、OKF のエコシステムが成熟した時点で改めて判断すれば十分です。

OKF が広く普及した場合

OKF のエコシステムが成熟し、型付きエッジの検索や検証つきの書き込みまでカバーするツール群が揃った場合、 選択肢は大きく 2 つあります。

A. このスキルを使い続ける
  • vault は OKF 適合なので、OKF 対応の閲覧ツールや可視化ツールをそのまま併用できる
  • 型付きエッジ・検証・自動書き戻しの仕組みはこのスキルが引き続き担う
  • 注意点: OKF 対応の別ツールで vault にファイルを追加した場合、graphrag-skill の型制約・エッジ検証・重複ゲート等の保護は効かない。手動追加と同じ扱いになる
B. OKF エコシステムに全面移行する
  • OKF ツール群が型定義・関係の検索・検証つき書き込みまでカバーするようになった場合、このスキルの役割はそちらに委譲できる
  • vault のファイルは OKF 適合なので、移行時にデータの変換は不要
  • vault 固有のフィールド（edges、aliases 等）は OKF の「未知フィールドを許容する」ルールで保持されるため、データが壊れることはない

いずれの場合も、フォーマットの互換性は既にあるので、将来の判断を急ぐ必要はありません。

§ 10

Token Capital — AI 時代の知識資産とこのスキルの位置づけ

汎用モデルを入れ替えても残る「ベテラン社員の頭の中」を、プレーンテキストで持つ。

2026 年 6 月、Microsoft CEO の Satya Nadella は Human Capital（人の知識・判断・パターン認識）と Token Capital（企業が構築・所有する AI 能力）の複利的成長を論じました。 要旨は 3 つです。

Learning Loop

人と AI の間に認知ループを作り、使うたびに改善するシステム。 改善されたワークフローがより良い信号を生み、暗黙知の蓄積を加速する複利構造。

Company Veteran テスト

汎用モデルを入れ替えても、その企業固有の「ベテラン的知識」は失われないこと。 モデルに依存しない知識の所有が問われる。

Hill Climbing Machine

IP としての学習ループ。他の資産と異なり複利的に蓄積し、 早く始めた企業ほど追いつかれにくい優位を持つ。

このスキルが本質的に一致する点

graphrag-skill はこのビジョンの語彙を借りずに、独立して同じ結論に到達しています。

学習ループの構造化

ask で引く → 作業する → add-* で書き戻す。 このサイクルが Hill Climbing Machine の具体実装です。 単なるログではなく Decision / RejectedOption / Risk という型で構造化されているため、 「回るだけ」ではなく「登り続ける」ループになります。

Company Veteran テスト合格

vault は Obsidian Markdown で、モデルに一切依存しません。 Claude を GPT に替えても Gemini に替えても、 「なぜこの設計にしたか」「なぜこの案を却下したか」は失われません。 ベテラン社員の頭の中をプレーンテキストで持つ設計です。

知識資産の自己所有

ランタイム依存ゼロ・ローカル完結。 SaaS に知識を預けず、自分のリポジトリ内で完結します。 少数モデルへの価値集中を避け、各チームが自分の学習ループを持つ構造です。

スケールと前提の違い

Nadella のビジョンは企業組織レベル（数千人規模の制度的記憶）を射程としており、 Private RL / Private Evals によるモデル自体の改善を含みます。 graphrag-skill はプロジェクト/リポジトリレベルで、 モデルを変えずコンテキスト（知識グラフ）を改善するアプローチです。

この違いは「合わない」のではなくレイヤーが違います。 Fine-tuning / RL は十分な利用規模がないと投資が回収できず、 小〜中規模チームではコンテキストの充実が正しい投資配分です。 逆に、このスキルが持つモデル非依存性（vault はプレーンテキスト）・検証可能性（全履歴が git で追え人間が読める）・耐障害性（クラウドが落ちても動く）は、 モデルの重みに知識を埋め込む方式では得にくい利点です。

組織スケールへの道 — focused vault + 横断検索

大きな組織で 1 つの vault に全知識をフラットに入れると、 「関係ない似たもの」がヒットします。 embedding の cosine は「意味が近い」を測りますが、「自分にとって関連がある」は測れません。 認証チームの Decision と決済チームの Decision は語彙が似ていても、互いの作業には関係がない。

自然なアーキテクチャは、チーム/ドメインごとの focused vault（日常の ask はここに閉じる）と、 必要な時だけ他の vault を覗く横断検索レイヤーです。 Nadella のビジョンをフラクタルに適用した構造で、 1000 人規模を 1 つの巨大 vault でなくフェデレーションで扱います。 既に world ヒント分離の実測（VAULT.md の lexical が 7/7 で支配的）がこの方向を裏付けています。

フィードバック — グラフのトポロジーを育てる

学習ループが複利的に回るには、フィードバック信号が要ります。 ただし、それは「ask が何回使われたか」という利用トレースではありません。

実際のフィードバックは、不具合や考慮漏れが見つかった時に判明します。 「本来 Concern/Component があって、そこから Constraint がエッジで繋がっていれば、 ask で引っ張れて防げた不具合だった」—— この「あるべきグラフ構造の不在」の検出がフィードバックの本質です。

このスキルは、不具合や見落とし発見時に 「グラフに適切な構造があれば回避できたか？ あるべきノードが存在しない状態か？」 を自問し、不足する Concern/Component とエッジを補うTopology Gap Review を備えています。 検索精度の改善ではなく、グラフのトポロジー自体を育てる行為が学習ループを閉じます。

graphrag-skill — GraphRAG knowledge base for Claude Code（v3: vault 単一正本、外部 graph DB なし）
正本スキーマ: graphrag/schema.ts スキル定義: skills/graphrag-knowledge/SKILL.md スキーマの根拠: references/carving-rationale.md
