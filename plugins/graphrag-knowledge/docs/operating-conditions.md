# 効きの前提条件 — このプラグインが機能する運用と、機能しなくなる運用

このプラグインは設計思想を持ちません。あなたのコードがどうあるべきかは言いません。この文書が並べるのはそれとは別のもの — **この道具が機能するための環境条件**です。冷蔵庫メーカーは「何を食べるべきか」は言いませんが「ドアは閉めてください」は言います。それと同じ位置づけで、守るかどうかはプロジェクトの自由ですが、**守らない場合にどの機能がどう死ぬか**は道具の側にしか言えないので、ここで先に言っておきます。

この文書に載る資格は3条件で判定しています (思想の混入を防ぐ選別器): **(i) 行為者がプロジェクトであり、(ii) 求める行為がこのプラグインの機構の使用・配線 (登記・エッジ・aliases・enforcer・verb 実行) であり、(iii) 死なせる機能を固有名で言える**こと。この試験を通らない一般的な開発助言は、たとえ正しくてもここには書きません (末尾の「解決しないこと」だけが例外で、あれは行為を求めない期待の切断です)。

各項目は、graphrag を約2ヶ月・のべ2プロジェクト (4ヶ月半/1971コミットのフル AI 開発と、1万行の E2E 基盤) で運用した実測に基づきます。

---

## 1. 知識は「配線するところまで」が書き込み

Decision や OperationalKnowledge を登記しても、対象ファイルへの `documented_by` / `constrains` / `sets_policy_for` エッジが無ければ、**読みの導線 (delta-check / area_map / frame-check) には一切乗りません**。権威を宣言する知識に、守りたい語彙 (シンボル名・リテラル) を `aliases` として入れなければ、authority echo (重複実装の現行犯検出) は発火しません。

- **死ぬ機能**: delta-check の connected_knowledge / authority_echoes、frame-map hook、area_map
- **実測**: 同型事故の運用知識が vault に在ったのに再発した事例は、知識がファイルに配線されていなかった。エッジがある知識は誤修正40件を止めた

## 2. Constraint には enforcer を書く (書けないなら正直に unenforceable)

散文の Constraint は、破られても何も落ちません。constraint-check が検査するのは**配線の腐敗だけ**で、違反そのものを検出するのはあなたのテスト・lint・型です。「破ったら落ちる検査」を書く気のない Constraint は、登記しても注意力頼みの飾りです — その場合は `--unenforceable "<理由>"` で「これは機械では守れない」と見え続ける形にしてください。

- **死ぬ機能**: enforcement contract 全体。unguarded が積もると enforcement_debt の警告自体が風景化する
- **実測**: Constraint 34件中23件が unguarded のまま数週間放置され、その領域で「one authority」原則の破れが49件蓄積した。逆に enforcer が居た場所では違反 commit が現行犯で止まった

## 3. 移行では旧側の撤去を Goal として同じ plan で登記する (Staged migration テンプレ)

プラグインは撤去残債を Goal として登記させ、その場所を触るたびに浮上させ、忘れられなくすることは**できます**。しかし登記されなかった残債を浮上させることはできず、「新側が動いたから移行完了」とプロジェクトが宣言することを覆すこともできません (完了の定義はあなたが決めるもので、このプラグインは踏み込みません)。分岐点は Staged migration テンプレ (撤去 Goal を移行と同じ plan で登記) を使うかどうか、それだけです。

- **死ぬ機能**: 撤去 Goal / debt-shadow の浮上装置 (登記されなければ浮上しようがない)
- **実測**: 最大の負債類型は「段階移行の旧側死に残り」。旧トポロジの部品が撤去されずに残り、死亡確認に大規模調査1本を要した

## 4. 定期精査イベントは人間が発注する

未登記の概念 (Concern) の発見や責務漏れの検出は、日常セッションには**構造的に起きません** (全 export を辿るようなコストを日常作業は払わない)。stocktake や pr-review は受け皿であって、発注者ではありません。定期的な精査を回すかどうか、いつ回すかは人間の仕事です。

- **死ぬ機能**: 登記が増えない → 1 の配線が痩せる → 読みの導線全体が痩せる、の連鎖
- **実測**: 1万行が Component 未登録のまま育ち、地図 (area_map) が全域で空振りしていた。精査1回で構造が登記され、以後の導線が機能し始めた

## 5. 大きな squash / 一発 merge の直後は、範囲全体に delta-check を回す

commit 境界の読みの導線は、commit の粒度でしか働けません。数百コミットを1つに畳むと、導線は一度しか発火せず、大塊の中の半端な配線は見出しに埋もれます。畳むこと自体は自由ですが、直後に `delta-check --diff <base>...<head> --full` を範囲全体へ1回 (`--full` = 通常の表示上限を外す)、が対になる運用です。

- **死ぬ機能**: commit 境界 hook の解像度
- **実測**: 195コミットの squash の中に、受信側だけ実装され送信側の無いコマンド配線が埋まり、3週間発見されなかった

## 6. 「あとで」と言ったら、その場で登記する文化

「別段階で」「Step 2 で」「あとで」— この言葉が出た瞬間が Goal (state: planned) の登記タイミングです。書き込みトリガは hook とスキルが毎回促しますが、**促しに応じる文化**はプロジェクト側にしか作れません。言った本人のセッションは消えます。登記されなかった「あとで」は、言われなかったのと同じです。

- **死ぬ機能**: open_goals (resume) / stale-goal 検出 (stocktake) / 残債の場所での浮上 (delta-check) — すべて登記が前提
- **実測**: コミットメッセージにだけ書かれた「全面的な権威委譲は別段階で」が2週間誰の視野にも入らず、その間 UI は嘘の完了表示を出し続けた

## このプラグインが解決しないこと (期待の切断)

上の6項目と違い、以下は何の行為も求めません — このプラグインに期待してはいけない領域の先回りの否認です。

- **検証環境の汚染** (手動配布・hotdeploy・実機とビルドの乖離)。知識も機械検査も、現実の側が嘘をつけば無力です。実測: パッケージへの資産同梱漏れを開発機の手動配布が隠し、同型事故の運用知識が vault に在ったのに誰も気づけなかった。これはナレッジ運用の外側の条件で、このプラグインのどの機能も代替しません。
- **初回の事故**。導線と免疫は「一度書き戻された種類の事故の2回目以降」を止める装置で、初見の失敗そのものは止めません。そこはテスト・レビュー・エージェントの完了検証の領分です。
- **登記されなかった知識**。エッジと語彙が張られていないものは、どの導線にも乗りません (clean は「配線された知識が無い」であって安全宣言ではない)。
